function call (not a special form), which is fine — `not` has no short-circuit
requirement. ✅

`local_max`. Empty `['and']` returns `True` (correct identity). ✅

**`or` (special form):** Returns first truthy value (Python-style, correct). Propagates
`local_max`. Empty `['or']` returns `False` (correct identity). ✅

**Whitelist:** `'not'`, `'and'`, `'or'` added to `_build_capability_env`. `not` is
the only one that needs it (the other two are special forms dispatched before symbol
lookup); the entries for `and`/`or` are harmless no-ops. ✅

**One semantic note (non-blocking):** `['and', True, 0]` returns `False`, not `0`.
Python's `and` would return `0`. This is a valid simplification for a boolean-focused
DSL. Document in `docs/REFERENCE.md` so future callers know `and` always returns a
Python `bool`, not the falsy operand.
